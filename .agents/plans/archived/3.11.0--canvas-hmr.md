# Canvas HMR Guard

## Problem

When editing prototype files during development, the canvas page re-renders via Vite HMR. This is disruptive because:
- The canvas has editing state (widget positions, selections, scroll position, zoom) that gets lost on re-render
- Canvas iframes (prototype embeds) already re-render on their own since they're separate browsing contexts
- The only time the canvas page itself should re-render is when working on the canvas UI code (in `packages/core` or `packages/react/src/canvas/`)

## Approach

Add a Vite plugin that intercepts HMR updates for canvas route modules and self-accepts them (no-op) unless a dev flag is active. The flag is a URL search param (`?hmr=1`) on the canvas page — easy to toggle when working on canvas UI code.

### How Vite HMR works for this case

1. A file changes → Vite finds affected modules in the module graph
2. For each module, Vite walks up the import chain to find a module that `accept`s the update
3. If no acceptor is found → full page reload
4. If an acceptor is found → the accepting module's callback runs (React Fast Refresh handles this for `.jsx` files)

### Strategy: `handleHotUpdate` in a Vite plugin

Add a new Vite plugin (`canvas-hmr-guard`) that:
1. Intercepts `handleHotUpdate` for all file changes
2. Checks if the current page is a canvas route (by checking connected WebSocket clients or a simpler heuristic)
3. If the changed file is NOT under `packages/react/src/canvas/` or `packages/core/src/` (canvas UI code), suppresses the update for canvas page modules

### Revised strategy: Client-side HMR guard

In `CanvasPage.jsx`, add `import.meta.hot.accept()` to self-accept HMR updates and make them a no-op. This means when any dependency of CanvasPage changes, React Fast Refresh won't force a re-render — the module just silently accepts and discards the update.

But this is too aggressive — it would also suppress canvas UI code changes.

### Final strategy: Vite plugin + URL flag

1. **Vite plugin** (`canvas-hmr-guard`): In `handleHotUpdate`, check if the changed module is canvas-related source code (`packages/react/src/canvas/` or `packages/core/`). If it is, let HMR proceed normally. If it isn't (e.g., prototype files, flow files, object files), send a custom event instead of the normal HMR update, effectively suppressing the re-render for canvas pages.

2. **URL flag override**: When the URL has `?hmr=1`, disable the guard entirely so all changes cause normal HMR behavior.

Wait — the problem is simpler than this. The canvas page doesn't import prototype files. Prototype files trigger full-reloads via the `prototypes-watcher` plugin and the data-plugin's `invalidate` function. The canvas route renders iframes for prototype content. So the issue is:

- When a data file (flow, object, record) or prototype page changes → the virtual module is invalidated → `full-reload` is sent → canvas page reloads
- When a prototype JSX file changes → React Fast Refresh handles it → but if it's not imported by the canvas, it shouldn't affect it anyway

The real culprit is `full-reload` events from:
1. `prototypes-watcher` plugin (add/unlink of prototype files)
2. `data-plugin` invalidating `virtual:storyboard-data-index` on data file changes
3. `storyboard-core/vite/server-plugin` on `toolbar.config.json` changes

### Actual strategy: Client-side full-reload suppression

The cleanest approach is:
1. In `CanvasPage.jsx`, register a `beforeunload`-style guard or use `import.meta.hot` to intercept incoming updates
2. Actually, Vite's `import.meta.hot` doesn't let you block full-reloads from the client side

### Actual strategy: Vite plugin that filters full-reload for canvas pages

The `canvas-reload-guard` plugin already suppresses full-reloads within a time window of canvas file mutations. We can extend this pattern:

**New plugin: `canvas-hmr-guard`** in `vite.config.js`:
1. Track whether the browser is on a canvas page via a custom WS message (canvas pages already dispatch `storyboard:canvas:mounted`)
2. When a `full-reload` would be sent, check if ALL connected clients are on canvas pages
3. If so, suppress the reload unless a `?canvas-hmr` flag is present in the URL

But Vite's server doesn't know the client URL... unless we add a client→server message.

### Simplest viable approach

Add a **client-side Vite plugin** that:
1. Listens for Vite's `vite:beforeFullReload` event on the client
2. If the current page is a canvas route and `?canvas-hmr` is NOT in the URL, prevents the reload
3. The iframes are unaffected because they're separate browsing contexts with their own Vite client

This is the cleanest solution — no server changes, no protocol extensions, just a small client-side guard.

## Steps

1. Add `canvas-hmr-guard` Vite plugin in `vite.config.js` that injects a client-side script
2. The script listens for `vite:beforeFullReload` and calls `event.preventDefault()` when on a canvas page (detected via `window.__storyboardCanvasBridgeState?.active`) and when `?canvas-hmr` is not in the URL
3. Verify iframes still HMR normally (they have separate Vite clients)
4. Test: edit a prototype file while on canvas → no reload. Add `?canvas-hmr` → reload works.

## Edge cases & risks

- **iframes must still update**: Since iframes load their own HTML with their own Vite client, they process HMR independently. The guard only affects the top-level document. ✅
- **Canvas UI changes need to work**: The `?canvas-hmr` flag provides an escape hatch. Could also detect if the changed file is in `packages/core/` or `packages/react/src/canvas/` and allow those through.
- **Multiple browser tabs**: Each tab has its own Vite client, so the guard applies per-tab based on whether that tab is showing a canvas page.
- **`vite:beforeFullReload` availability**: This is a standard Vite client event, available since Vite 2.x.
