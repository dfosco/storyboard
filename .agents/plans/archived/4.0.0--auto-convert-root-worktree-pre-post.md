# `sb dev` root-to-worktree conversion — Pre/Post Architecture

## Before (current behavior)

### How `sb dev` resolves where to run

When you run `sb dev` with **no arguments**, the code does one thing:

```js
return { worktreeName: detectWorktreeName(), targetCwd: process.cwd(), created: false }
```

`detectWorktreeName()` checks (in order):
1. Is cwd inside `.worktrees/<name>/`? → return that name
2. Is the current git branch not main/master? → return the branch name
3. Otherwise → return `'main'`

The **worktree name** drives two things:
- The base path: `'main'` → `/`, anything else → `/branch--<name>/`
- The port: `'main'` → 1234, anything else → 1235+

But `targetCwd` is **always `process.cwd()`** — the directory you ran the command from.

### What this means in practice

| Scenario | worktreeName | targetCwd | basePath | Result |
|----------|-------------|-----------|----------|--------|
| Root on `main` | `main` | repo root | `/` | ✅ Correct |
| Inside `.worktrees/foo/` | `foo` | `.worktrees/foo/` | `/branch--foo/` | ✅ Correct |
| Root on branch `feature-x` | `feature-x` | repo root | `/branch--feature-x/` | ⚠️ Works, but root stays on non-main branch |

### Pros of current behavior
- **Zero magic** — detect name, use cwd
- **No git mutations** — never touches your branch or creates directories
- **Fast** — no npm install, no worktree creation
- **Safe with dirty trees** — doesn't care about uncommitted changes

### Cons of current behavior
- Root can get "stuck" on a non-main branch indefinitely
- No actual worktree isolation (root is shared state)
- Inconsistent with the worktree model (name says branch, directory says root)

## After (proposed behavior)

### Changes

1. **`sb dev` prompts** — when root is on a non-main branch, asks the user before converting
2. **`sb code [branch]`** — new command to open a worktree in VS Code
3. **`sb setup`** — now installs the VS Code `code` CLI

### `sb dev` flow (no arguments, root on non-main branch)

```
sb dev (from repo root, branch = feature-x)
  │
  ├─ detectWorktreeName() → "feature-x"
  ├─ isAtRoot? → yes
  │
  ├─ worktree exists? → yes → use existing worktree, done
  │                   → no  ↓
  │
  ├─ PROMPT: Convert "feature-x" to a worktree? [Y/n]
  │     │
  │     ├─ No/Cancel → proceed with root as-is (legacy behavior)
  │     │
  │     └─ Yes:
  │         ├─ check: uncommitted changes? → abort
  │         ├─ check: default branch exists? → abort if not
  │         ├─ git checkout main
  │         ├─ git worktree add .worktrees/feature-x feature-x
  │         ├─ npm install
  │         │
  │         ├─ PROMPT: Open in VS Code? [Y/n]
  │         │     └─ Yes → code .worktrees/feature-x
  │         │
  │         └─ start Vite from .worktrees/feature-x/
```

### `sb code [branch]`

```
sb code              → opens current worktree or repo root
sb code main         → opens repo root
sb code feature-x    → opens .worktrees/feature-x/
```

Errors helpfully if worktree doesn't exist (lists available ones) or if `code` CLI is not installed.

### `sb setup` changes

Adds step 5 (before proxy): checks for VS Code `code` CLI. If not in PATH but VS Code.app is installed, symlinks it to `/usr/local/bin/code`. Otherwise prints manual instructions.

### Pros
- **User stays in control** — prompt means no surprises
- **Declining is safe** — falls back to existing behavior
- **`sb code` is independently useful** — quick worktree switching in VS Code
- **`sb setup` handles toolchain** — `code` CLI is a reasonable dev dep

### Cons
- **Interactive prompt blocks non-TTY / CI** — but `sb dev` is inherently interactive
- **Conversion still mutates git state** — but only after explicit consent
- **Two prompts back-to-back** — convert prompt + VS Code prompt could feel chatty

| | Before | After |
|---|--------|-------|
| Lines in `resolveDevTarget` no-arg path | 1 | ~35 |
| Git mutations | 0 | 1 (only with user consent) |
| New files | 0 | 2 (`code.js`, `dev-helpers.js`) |
| New CLI commands | 0 | 1 (`sb code`) |
| User prompts | 0 | 2 (convert? + VS Code?) |
| Behavior if user declines | n/a | Same as before |
