# CLI Create Expansion

## Problem

The `storyboard create` CLI is interactive-only. Users and AI agents can't script creation of prototypes, canvases, flows, pages, or canvas widgets without going through prompts. The UI already supports all these via HTTP POST endpoints — the CLI just needs flag-based alternatives.

## Approach

1. **Extract a shared flag parser** — a small `parseFlags(argv)` utility that converts `--key value` and `--bool` args into an object. Keeps the CLI dependency-free (no yargs/commander).
2. **Add flag support to existing commands** — `create prototype` and `create canvas` accept all fields as `--flags`. If all required flags are provided, skip prompts entirely. If some are missing, prompt only for those.
3. **Add new subcommands** — `create flow`, `create page`, `canvas add <widget-type>`.
4. **Define JSON schemas** for each creation type in a `schemas/` directory, so future tools can validate and extend.
5. **Update the help screen** to document all flags.

## Endpoints (already exist on dev server)

| Command | Endpoint | Required params |
|---------|----------|----------------|
| `create prototype` | `POST /_storyboard/workshop/prototypes` | `name` |
| `create canvas` | `POST /_storyboard/canvas/create` | `name` |
| `create flow` | `POST /_storyboard/workshop/flows` | `name`, `prototype` |
| `create page` | `POST /_storyboard/workshop/pages` | `prototype`, `path` |
| `canvas add` | `POST /_storyboard/canvas/widget` | `name` (canvas), `type` |

## Files to change/create

### New files
- `packages/core/src/cli/flags.js` — shared flag parser + schema validation
- `packages/core/src/cli/schemas.js` — creation schemas (prototype, canvas, flow, page, widget)
- `packages/core/src/cli/createFlow.js` — `create flow` subcommand
- `packages/core/src/cli/createPage.js` — `create page` subcommand  
- `packages/core/src/cli/canvasAdd.js` — `canvas add <type>` subcommand

### Modified files
- `packages/core/src/cli/index.js` — register new commands (`create flow`, `create page`, `canvas add`)
- `packages/core/src/cli/create.js` — integrate flag parser, skip prompts when flags provided

## Steps

### 1. Create `flags.js` — shared flag parser
- Parse `--key value`, `--key=value`, `--bool-flag` from argv
- Convert `--no-grid` → `{ grid: false }`
- Export `parseFlags(argv, schema)` that returns `{ flags, missing, errors }`
- Schema defines: name, type (string/boolean/number/array), required, default, description

### 2. Create `schemas.js` — creation schemas
- Export schema objects for: prototype, canvas, flow, page, widget
- Each schema entry: `{ type, required, default, description, aliases }`
- Schemas serve as both validation and documentation source

### 3. Refactor `create.js` — flag support for prototype & canvas
- Parse flags from argv using `parseFlags`
- If all required fields provided via flags → non-interactive POST
- If some missing → prompt only for missing fields
- Existing interactive flow unchanged when no flags passed

### 4. Create `createFlow.js` — flow creation subcommand
- Flags: `--name`, `--prototype`, `--folder`, `--globals`, `--copy-from`, `--author`, `--description`
- Interactive fallback for missing required fields
- POSTs to `/_storyboard/workshop/flows`

### 5. Create `createPage.js` — page creation subcommand
- Flags: `--prototype`, `--path`, `--folder`, `--template`
- Interactive fallback for missing required fields
- POSTs to `/_storyboard/workshop/pages`

### 6. Create `canvasAdd.js` — canvas widget addition
- Usage: `storyboard canvas add <type> --canvas <name> [--x N] [--y N] [--props '{}']`
- POSTs to `/_storyboard/canvas/widget`
- Known types: `sticky-note`, `markdown`, `prototype`

### 7. Update `index.js` — register new commands
- Add `create flow`, `create page` to the create dispatcher
- Add `canvas add` as a new top-level command
- Update help screen with new commands and flag documentation

## Edge cases & risks

- **Dev server must be running** for all creation commands — existing `ensureDevServer()` handles this
- **Flag parsing ambiguity** — `--name my proto` should only consume one word; multi-word values need quotes
- **Backwards compatibility** — no-flag usage must remain identical to current behavior
- **Schema extensibility** — new creation types can be added by just adding a schema + handler
