# Fix: Caddy proxy duplicate-route cleanup

## Problem

When Caddy is started or reloaded from a **Caddyfile**, it creates route entries without an `@id` and with `terminal: true`. Later, when `storyboard dev` runs and calls `upsertCaddyRoute()` via the admin API, the PATCH-by-`@id` fails (since the Caddyfile route has no `@id`), so a **POST** appends a new route with an `@id`. The result: two routes for the same host — and the Caddyfile-sourced one (ordered first, marked `terminal`) catches all traffic before the admin-API route can serve branch-specific paths.

**Symptoms:** `http://<domain>/branch--<name>/` returns 502 or falls through to the main port instead of the worktree port, while `http://localhost:<port>/branch--<name>/` works fine.

## Approach

Add a **cleanup step** in `upsertCaddyRoute()` that removes duplicate routes for the same host after a successful upsert. This handles both POST (new route) and PATCH (update) scenarios — a stale Caddyfile route could exist in either case.

## Files to change

- `packages/core/src/cli/proxy.js` — add `cleanupDuplicateRoutes()` helper, call it after PATCH and POST

## Steps

1. **Add `cleanupDuplicateRoutes(keepId, host)`** — queries Caddy admin API for all routes, finds ones matching the same host but with a different (or no) `@id`, deletes them from highest index to lowest (to preserve indices during deletion).
2. **Call cleanup after successful PATCH** — a stale Caddyfile route could still exist even when our `@id` route is already present.
3. **Call cleanup after successful POST** — this is the primary scenario: PATCH failed because no `@id` existed, POST added a new route, but the old Caddyfile route is still there.

## Edge cases & risks

- **Index shifting during deletion**: Deleting routes by index from highest to lowest avoids this.
- **Concurrent dev servers**: Two `storyboard dev` processes starting simultaneously could race on cleanup. This is best-effort and extremely unlikely in practice.
- **Cleanup failure**: Wrapped in try/catch — if it fails, the upsert still succeeds. The worst case is the status quo (duplicate routes).
- **Multiple repos sharing a domain**: Not possible — each repo has its own `devDomain` in `storyboard.config.json`.
